大数据概览

什么是大数据

几乎所知的所有的大数据应用，其根本的目的都是为了解决两个基本问题——海量数据的存储和海量数据的计算，其核心思想都是分布式，但其跟现在大热的区块链的完全去中心化的分布式有着一定的区别，在区块链的概念和体系中所有节点的角色和地位都是一样的，但是大数据的分布式体系中，角色的定义和地位是不一样的，他们都是一种Master和Slaves的模式。

Hadoop

一提到大数据，Hadoop就不得不提，Hadoop有两大核心，HDFS和MR,前者是一个分布式文件存储系统，全称是Hadoop Distribute File System，用来解决海量数据储存；MR即Map和Reduce算法框架，用来解决海量数据的计算问题。先来看HDFS，这是整个Hadoop建筑的地基。它其实是将一个大文件分成若干块保存在不同服务器的多个节点中。通过联网让用户感觉像是在本地一样查看文件，为了降低文件丢失造成的错误，它会为每个小文件复制多个副本（默认为三个），以此来实现多机器上的多用户分享文件和存储空间。

HDFS的特点：

  • 保存多个副本(默认是三个)，且提供容错机制，副本丢失或者宕机自动恢复
  • 运行在多个廉价的机器上
  • 适合大数据的处理。因为小文件也占用一个块，小文件越多（1000个1k文件）块越 多，NameNode压力越大。

HDFS基本架构：

从图中可以看出HDFS基本结构分为NameNode、SecondaryNameNode，DataNode三个。

NameNode： Master节点，类似于Linux中的根目录。管理数据块映射，处理客户读写请求，配置副本策略，管理HDFS名称空间。

SecondaryNameNode：NameNode的冷备份；保存NameNode的信息（不是全部信息，用于NameNode宕机之后数据的恢复）

DataNode：负责文件数据的实际物理存储

fsimage：元数据镜像文件（文件系统的目录树）

edits：元数据的操作日志

NameNode=fsimage+edits

NameNode详解

NameNode是整个HDFS的中心点，用户所有对HDFS的读写请求都需要经过NameNode的处理

NameNode包含：目录与数据块之间的关系（靠fsimage和edits实现），数据块和节点之间的关系。需要注意的是NameNode中仅仅存储的是目录树的信息，而像Block与节点之间的关系则是通过DataNode上传的。

DataNode

提供真实文件数据存储服务，根据需要（客户端或者NameNode的调度）检索或者存储数据块。

数据块(block)是整个HDFS中最基本的数据存储单位，Hadoop1中默认大小是64M，Hadoop2中是128M。对于文件内容而言，一个文件的大小是size，按照固定的大小，从0开始，顺序给文件编号划分，划分好的每一块就称之为一个block。与普通的文件系统不同的是，HDFS中，如果一个文件小于一个数据块的大小，并不占用整个数据块存储空间。
(这样设置可以减轻namenode压力，因为namonode维护者文件与数据块列表的对应大小)

HDFS写流程

流程图：

有一个文件FileA ，100M大小。Client将FileA写入到HDFS上。

HDFS按默认配置。

HDFS分布在三个机架上Rack1，Rack2，Rack3。

  • Client端将FileA按照64M进行分块，分成block1和block2两块
  • Client向NameNode发送写数据请求，如图蓝色虚线①——>。

  • NameNode节点记录block信息，并且返回可用的DataNode节点，如粉色虚线②———>。
    Block1: host2,host1,host3

    Block2: host7,host8,host4

    原理：

    NameNode具有RackAware机架感知功能，这个可以配置。

    如果Client是DataNode节点，则block第一个副本存储在Client所在的DataNode上，第二个副本存储在与第一个block不同机架的DataNode上，第三个副本与第二个副本在同一个机架但不同的DataNode上。

    如果Client不是DataNode节点，那么副本1就是随机选择的，副本2与副本3的选择依据与第一种情况一致。

  • Client端开始想HDFS写入数据,整个过程是流失写入的：

    • 将64M的block1按64k的package划分;
    • 将第一个package发送给host2;
    • host2接收完成之后，开始将第一个package发送给host1，同时Client开始向host2发送第二个package；
    • host1接收完第一个package之后，开始向host3发送第一个package,同时开始接收host2发送过来的package；
    • 一次类推，直至block1的数据发送完成，如图红线实线所示；
    • host2,host1,host3向NameNode，host2向Client发送通知，说“消息发送完了”。如图粉红颜色实线所示。
    • client收到host2发来的消息后，向namenode发送消息，说我写完了。这样就真完成了。如图黄色粗实线
    • 发送完block1后，再向host7，host8，host4发送block2，如图蓝色实线所示。
    • 发送完block2后，host7,host8,host4向NameNode，host7向Client发送通知，如图浅绿色实线所示。
    • client向NameNode发送消息，说我写完了，如图黄色粗实线。。。这样就完毕了。

分析整个写过程，我们可知：

  • HDFS实际占的空间是数据文件实际大小的3倍，所需要的带宽也是三倍
  • 在整个写数据的过程中，NameNode与DataNode通过心跳机制保持相互通信,从而让NameNode确认DataNode还活着，如果某个DataNode挂掉可以将要写入挂掉DataNode的数据放到其他节点上去。

HDFS读流程

读的流程就相对简单些,如图所示，client要从datanode上，读取FileA。而FileA由block1和block2组成。
那么，读操作流程为：

a. client向namenode发送读请求。

b. namenode查看Metadata信息，返回fileA的block的位置。

block2:host7,host8,host4

c. block的位置是有先后顺序的，先读block1，再读block2。而且block1去host2上读取；然后block2，去host7上读取；

MapReduce

先上一个整体的流程图

让我们从头开始慢慢梳理，整个MapReduce的所有过程。

Input Split（输入分/切片）

对于一个MapReduce算法，在开始map计算之前,MapReduce会根据文件的大小来计算输入分片(input split)，每一个输入分片实际对应的是一个map任务，输入分片记录的并不是存储的数据本身，而是一个分片长度和一个记录数据的位置的数组。

那么每个分片的大小和分片的个数怎么确定呢？

hadooop提供了一个设置map个数的参数mapred.map.tasks，我们可以通过这个参数来控制map的个数。但是通过这种方式设置map的个数，并不是每次都有效的。原因是mapred.map.tasks只是一个hadoop的参考数值，最终map的个数，还取决于其他的因素。

为了方便介绍，先来看几个名词：

block_size : hdfs的文件块大小，默认为64M，可以通过参数dfs.block.size设置

total_size : 输入文件整体的大小

input_file_num : 输入文件的个数

默认map个数

如果不进行任何设置，默认的map个数是和blcok_size相关的。

default_num = total_size / block_size;

期望的大小

可以通过参数mapred.map.tasks来设置程序员期望的map个数，但是这个个数只有在大于default_num的时候，才会生效。

goal_num = mapred.map.tasks;

设置处理的文件大小

可以通过mapred.min.split.size 设置每个task处理的文件大小，但是这个大小只有在大于block_size的时候才会生效。

split_size = max(mapred.min.split.size, block_size);
split_num = total_size / split_size;

计算的map个数

compute_map_num = min(split_num,  max(default_num, goal_num))

除了这些配置以外，mapreduce还要遵循一些原则。 mapreduce的每一个map处理的数据是不能跨越文件的，也就是说min_map_num >= input_file_num。 所以，最终的map个数应该为：

final_map_num = max(compute_map_num, input_file_num)

Map阶段（包含shuffle）

Input Split 完成将会确定map的个数，即整个mr任务中起始的MapperTask的数量。

  • 对输入片中的记录按照一定的规则解析成键值对。有个默认规则是把每一行文本内容解析成键值对。“键”是每一行的起始位置(单位是字节)，“值”是本行的文本内容，即.

  • MapperTask调用Map方法,将处理成,这个Map函数是整个MR过程中真正需要程序员自己去实现的逻辑方法。这个map的输出结果会放先在一个环形内存缓冲区中(默认大小是100M，由mapreduce.task.io.sort.mb属性控制)，并做一次预排序，当缓冲区快要溢出时（默认为缓冲区大小的80%，由mapreduce.map.sort.spill.percent属性控制），后台程序做以下几件事：

    1.线程会跟据数据最终要传达的到Reducer把缓冲区划分(默认按照键)相应的分区.
    partition的目是将记录划分到不同的Reducer上去，以期望能够达到负载均衡，以后的Reducer就会根据partition来读取自己对应的数据

    2.二次排序，首先根据数据所属的partition进行排序，然后每个partition中的数据再按key来排序.

    3.运行combiner(如果设置了的话)，combiner的本质也是一个Reducer，其目的是对将要写入到磁盘上的文件先进行一次处理，这样，写入到磁盘的数据量就会减少。

    4.将数据写到本地磁盘产生spill文件(spill文件保存在{mapred.local.dir}指定的目录中，Map任务结束后就会被删除)

最后，每个Map任务可能产生多个spill文件，在每个Map任务完成前，会通过多路归并算法将这些spill文件归并成一个文件。至此，Map的shuffle过程就结束了。

Reduce阶段(包含shuffle)

1.Copy阶段：Reducer通过Http方式得到输出文件的分区。
    首先要将Map端产生的输出文件拷贝到Reduce端，但每个Reducer如何知道自己应该处理哪些数据呢？因为Map端进行partition的时候，实际上就相当于指定了每个Reducer要处理的数据(partition就对应了Reducer)，所以Reducer在拷贝数据的时候只需拷贝与自己对应的partition中的数据即可。每个Reducer会处理一个或者多个partition，但需要先将自己对应的partition中的数据从每个Map的输出结果中拷贝过来。

2.sort阶段，也称为merge阶段。
    复制数据的时候，把可以存放进内存的就把数据存放在内存中，当达到一定的时候，启动merge，将他们再次排序、分组合并，最后形成一个最终文件，作为Reduce任务的输入。此时标志Shuffle阶段结束，然后Reduce任务启动，将最终文件中的数据处理形成新的键值对<k3,v3>。
